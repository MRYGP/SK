# AI拆解系统架构 · 从"人读知识库"到"AI驱动的商业诊断系统"

> 版本：v0.1（架构蓝图，非最终实施方案） 创建日期：2026-03-27 触发事件：与Kimi的多维度商业分析对话 + 内部推演 状态：🔮 架构设计阶段，未开始实施 存储位置：meta/AI拆解系统架构-v0.1.md

------

## 一、为什么要做这件事

### 当前状态

SK仓库是**给人看的知识库**：

- 理论文档是叙事性的，人读起来舒服，但AI没法从中自动抽取判断维度
- 案例是故事性的，人读了有启发，但AI没法拿两个案例在同一个维度上精确比对
- 模板是流程性的，告诉人"先做什么再做什么"，但AI需要结构化的输入输出定义

### 目标状态

让SK仓库同时是**人的工作台**和**AI的大脑**。

类比：不是ChatGPT那种全科门诊，而是一个**商业机会诊断的专科医生**—— 有自己的病例库（案例）、诊断手册（维度注册表）、检查流程（拆解模板）、会诊机制（跨维度矛盾检测），还知道什么时候要转诊（触发深度研究）。

### 核心价值

人拆解一个产品，实际深入的维度大概3-4个，其余靠直觉补位。 这不是能力问题，是认知带宽的硬约束。

AI可以在40个维度上保持均匀的分析深度。 但AI没有"这事靠不靠谱"的嗅觉，没有权重判断的体感，生成不了反直觉洞察。

所以最优解不是"AI替代人"，而是**重新分工**：

- 广度靠AI，深度靠人
- 初筛靠AI，校准靠人
- 案例匹配靠AI，嗅觉判断靠人

### 第一层：维度层 · dimension-registry.yml

**它是什么：** AI分析任何产品时的"检查清单"。把大师们的理论拆成一条条可独立执行的判断维度。

**为什么需要它：** 克里斯坦森看产品，本能看"它在颠覆谁"。蒂尔看同一个产品，看"有没有垄断特征"。波特看五种竞争力量。他们每人只用1-2个维度看到极致。没有人能同时用所有维度看同一个产品。AI可以。

**每条维度的格式：**

```yaml
dim_id: "DIM-001"
dim_name: "破坏性定位"
source_theory: "DOC-S032-破坏式创新"
source_author: "克里斯坦森"
category: "竞争"  # 需求/模式/竞争/团队/时机/增长/防御
core_question: "这个产品是从低端切入还是正面进攻在位者？"
judgment_criteria:
  healthy: "从在位者看不上的市场切入，性能够用即可"
  warning: "正面进攻但有差异化角度"
  danger: "同质化竞争，跟在位者在同一个维度比性能"
related_dims: ["DIM-002", "DIM-003"]  # 关联维度
typical_cases:
  success: "Netflix从DVD邮寄切入（在位者看不上）"
  failure: "某AI搜索引擎正面硬刚Google"
```

**维度来源盘点（基于现有知识库）：**

| 来源理论                                        | 可提取维度数（估） | 维度类别       |
| ----------------------------------------------- | ------------------ | -------------- |
| DOC-S032 破坏式创新                             | 4-5个              | 竞争           |
| DOC-S037 从0到1                                 | 4-5个              | 防御/竞争      |
| DOC-S047 三大标杆方法论                         | 3个                | 竞争/团队      |
| DOC-S061 网络效应与学习效应                     | 3个                | 增长/防御      |
| DOC-S065 深度用户护城河                         | 3个                | 增长/防御      |
| DOC-D021 维度跨越                               | 2个                | 竞争（元维度） |
| DOC-S074 换题思维                               | 2个                | 需求（元维度） |
| DOC-D022 RRF判断链                              | 3个                | 模式/增长      |
| DOC-P001 财富公式                               | 3个                | 增长           |
| DOC-P002 推演公式                               | 2个                | 时机           |
| failure_modes.yml（FM001-FM015）                | 15个               | 风控           |
| 天时（技术周期/政策/资本/情绪）                 | 4个                | 时机           |
| 地利（中国市场/基础设施/供应链/区域差异）       | 4个                | 时机/需求      |
| 人和（创始人匹配度/团队半径/合伙人缺口/生态网） | 4个                | 团队           |
| **合计（估）**                                  | **~55个**          |                |

> 注意：55个是上限估计。实际提取时会有重叠和合并，预计最终30-40个有效维度。

**人和AI的分工：**

- AI：对每个维度逐条跑判断（healthy/warning/danger），输出全景扫描报告
- 人：看完40个维度的扫描结果后，判断"此刻哪3个维度最关键"，集中深挖

```yaml
# ========== 新增：维度级标签（用于跨案例同维度比对） ==========

  - dim_id: "DIM-001"
    dim_name: "破坏性定位"

    note: "从在位者看不上的低端市场切入"
  - dim_id: "DIM-007"
    dim_name: "网络效应类型"
    performance: "failure"
    note: "试图建双边市场但冷启动失败"
  # 不用标全部40个维度，只标最突出的5-8个

# ========== 新增：关键分叉点（用于场景F快速诊断） ==========
pivot_point:
  decision: "2024年3月选择正面进攻企业市场，放弃个人用户"
  alternative: "如果先做个人免费版再自下而上渗透"
  outcome_assessment: "这个选择直接导致获客成本飙升"
  category: "战场选择"  # 战场选择/商业模式/团队/时机/技术路径
```

**升级后能做什么（现有格式做不到的）：**

- 搜"DIM-007网络效应类型 + performance=failure" → 所有在网络效应上栽跟头的案例
- 搜"category=战场选择 + outcome_assessment含'失败'" → 所有战场选错的分叉点
- 两个案例在同一个维度上做精确比对，而不是整体比较

### 第三层：模式层 · patterns.yml（全新）

**它是什么：** AI从案例库中发现的跨案例规律。不是人写的理论，是从数据中涌现的。

**触发条件：** 每积累5-10个新案例，跑一次模式提取。

**格式示例：**

```yaml
pattern_id: "PAT-001"
discovered_date: "2026-XX-XX"
discovery_method: "跨案例分析（N=15）"
status: "待验证"  # 待验证 / 已验证 / 已否定

pattern_statement: >
  选择低端切入（DIM-001=healthy）+ 团队缺乏行业经验（DIM-031=danger）
  的产品，3年内失败率极高（8/10案例失败）。
  
  推测机制：低端切入需要对行业运转细节有深刻理解才能找到
  "够用即可"的性能阈值，外行团队容易把"够用"定得太低或太高。

supporting_cases: ["CASE-003", "CASE-007", "CASE-012"]
contradicting_cases: ["CASE-009"]  # 反例
confidence: "C"  # A/B/C/X，沿用现有证据等级
human_review: "待审核"  # 待审核 / 杨确认 / 已否定
action_implication: "如果看到低端切入+外行团队组合，优先检查行业认知深度"
```

**关键规则：**

- 模式必须经过人审核才能标为"已验证"
- AI发现的模式可能是真洞察，也可能是幸存者偏差
- "已验证"的模式才能进入拆解流程作为参考
- 每个模式必须有反例字段（contradicting_cases），防止过度拟合

## 三、两个关键循环

### 循环一：研究触发循环（"什么时候去查"）

现有的深度研究触发规则是"数据有空白就提醒"。升级为维度级的精准触发：

```
AI跑完40维度初筛
    ↓
每个维度自动标注证据等级（A/B/C/X）
    ↓
计算该维度对当前产品的"权重"（基于品类+阶段）
    ↓
如果：高权重维度（>20%影响）+ 低证据（C或X级）
    → 触发定向深度研究，指定维度和关键问题
    ↓
如果：低权重维度 + 低证据
    → 不触发，标注"数据不足但影响有限"
```

**与现有工具的接口：**

- 触发后，提醒用户用深度研究员跑对应模式
- 研究员返回数据后，更新该维度的证据等级
- 重新跑维度扫描，看结论是否变化

### 循环二：知识回流循环（"怎么越跑越强"）

```
每次完成一个拆解
    ↓
① 新案例卡入库（带维度标签+分叉点）
    ↓
② 检查是否有新维度需要补充到注册表
   （拆解中发现了某个理论维度注册表里没有的判断角度）
    ↓
③ 每5个案例 → 跑一次模式提取
   AI扫描案例卡库，识别跨案例规律
    ↓
④ 人审核模式 → 确认/否定
    ↓
⑤ 已验证模式 → 进入下一次拆解的参考库
```

**复利效应：** 第1个案例拆解时，AI只有维度注册表（理论维度）。到第50个案例，AI还有模式库（实战规律）+ 丰富的维度级案例比对库。第100个案例的拆解质量远好于第1个。

## 四、人和AI的分工矩阵

| 环节             | AI做什么                           | 人做什么                               |
| ---------------- | ---------------------------------- | -------------------------------------- |
| **维度扫描**     | 40个维度逐条跑判断，输出全景报告   | 判断"此刻哪3个维度最关键"              |
| **案例匹配**     | 在案例库中按维度精确检索相似案例   | 判断匹配结果是否真的相似               |
| **矛盾检测**     | 自动识别"维度A结论和维度B结论打架" | 判断这个矛盾意味着什么                 |
| **模式识别**     | 从案例库中发现跨案例规律           | 审核模式是真洞察还是幸存者偏差         |
| **权重判断**     | 提供默认权重（基于品类+阶段）      | 根据体感校准实际权重                   |
| **嗅觉判断**     | 做不了                             | 人的核心不可替代价值                   |
| **反直觉洞察**   | 推荐候选（基于维度矛盾）           | 从候选中提炼真正反直觉的那个           |
| **中国机会扫描** | 搜索公开数据                       | 补充地面情报和人脉信息                 |
| **证据标注**     | 自动标注A/B/C/X                    | 校准（"这个X级根据我的经验其实是B级"） |
| **文章写作**     | 生成初稿框架                       | 注入个人风格和锋利度                   |

## 六、与"活书"方向的关系

这个架构直接支撑"活书"的杀手级应用：

**场景：** 读者看完一篇拆解文章，文章底部有入口——"输入你关注的产品，用同样的框架自己跑一遍"。

**技术路径：** 小程序调用维度注册表 → AI跑40维度初筛 → 自动匹配案例库 → 输出初步诊断报告 → 读者根据自己的判断校准。

**内容从"读"变成"用"，就在这一步实现。**

这也是吸引合伙人的最强信号——展示的不是"一个人的洞察力"，而是"一个可复制、可扩展的能力基础设施"。

## 八、风险与约束

| 风险                        | 应对                                                         |
| --------------------------- | ------------------------------------------------------------ |
| 维度太多导致分析瘫痪        | 严格执行"AI广扫→人聚焦3个"的分工                             |
| 模式库被过度拟合历史        | 每个模式必须有反例字段+人工审核                              |
| 架构建设耽误内容发布        | **行为原则：出现 > 优化**。架构建设穿插在日常拆解中，不单独占用时间 |
| 维度注册表维护成本          | 初始建设后增量很小，主要是新理论入库时才需要更新             |
| 当前只有1张案例卡，数据不够 | Phase 0-1都是在"正常拆解工作中顺便积累"，不是专门为架构做事  |

## 十、这件事的本质

回到Kimi对话的原始洞察——

市场上缺的不是"更多框架"，而是**"框架之间的翻译器和迭代引擎"**。

dimension-registry.yml就是翻译器：把大师们各自的语言翻译成统一的结构化维度。 patterns.yml就是迭代引擎：让系统从每次拆解中学到新东西。 case-cards的维度标签就是索引：让AI能在案例之间做精确的维度级比对。

三者合在一起，让(人+AI)^n的复利效应真正落地。
