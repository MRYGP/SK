## 三、为什么这件事重要

**对 Claude(主对话)的意义**:
- 不再需要承担"头发产品每天执行细节",可以专注**战略 / 方法论 / 跨方向 / 哨兵**

- GitHub 现读保证状态与 SK 主仓库同步

**对杨(用户)的意义**:
- 战略 / 方法论问题 → 找 Claude(主对话)
- 头发产品执行 / 卖家访谈 / Demo 验收 → 找 ChatGPT Project
- 单点工具(写作 / 第一读者 / 深度研究) → 找对应 GPTs

## 五、两层之间的同步机制

### 5.1·SK → SKGPT 方向

当 SK 主仓库的关键文件更新(BP / 任务书 / 推演流水账)时:
- 杨手动同步到 `D:\skgpt\project-upload\`
- 重新上传到 ChatGPT Project(或让 Project 通过 GitHub 现读拿到最新)

### 5.2·SKGPT → SK 方向

**这是关键也是弱点**——ChatGPT Project 的反馈不会自动入 SK 库:

- 业内访谈反馈 / 卖家真实数据 / Demo 验收结果 → 杨手动整理回 `D:\sk\cases\2026\...`
- 元层教训(哨兵触发、方法论修正) → 杨手动整理回 `D:\sk\ops\清单草稿-未验证教训.md`
- 关键判断 / 漂移记录 → 杨手动整理回 `D:\sk\ops\执行状态总表.md`

**这件事靠杨手动**——这是当前设计的弱点,但没有更好方案。

### 5.3·GitHub 现读协议

ChatGPT Project 走 GitHub 现读时,优先读 SK 主仓库的 `main` 分支:
- 当前阶段 → `ops/执行状态总表.md`
- 头发主线 → `cases/2026/推演流水账/推演流水账-头发产品-2026-05-15/README.md`
- 哨兵触发 → `ops/清单草稿-未验证教训.md`

详见:`D:\skgpt\protocols\github-read-protocol.md`

## 六、Claude 在分层后该怎么做

### Claude 的零号动作不变

- `list_allowed_directories` 确认 Filesystem 可用
- 读 `ops/执行状态总表.md`(已加 SKGPT 引用)

### Claude 的一号动作新增

如果**用户问到头发产品的具体执行细节**(Demo 进度 / 卖家访谈反馈 / Project 内的讨论):
- **第一反应**:这件事属于 ChatGPT Project 范围,不是 SK 主对话范围
- 提醒用户:"这件事建议在 ChatGPT Project 内讨论。SK 主对话(Claude)负责战略和方法论。"
- 例外:用户明确说"我要和你讨论 / 我要入 SK 库 / 我要做跨方向判断"

### Claude 的职责仍然是

- 战略层判断(铁律 / 北极星 / 路径)
- 方法论沉淀(SKILL / 理论 / 元教训)
- 跨方向对比(头发 vs 其他产品)
- 哨兵机制(漂移检测)
- 仓库维护(执行状态总表 / 清单草稿)
- 公众号内容(其他线)

### Claude 不该做的事(新增边界)

- 不要在 SK 主对话里详细推进头发产品技术细节(去 Project)
- 不要在 SK 主对话里替代业内访谈结论(等 5/22 真实数据)
- 不要把 Project 内的讨论自动入 SK 库(等杨手动同步)

## 七、未来扩展空间

这套分层结构**不只服务头发产品**。未来 SK 做下一个产品时,可以按同样模式建:
- 新 ChatGPT Project(在 SKGPT 仓库下加新 projects/ 文件)
- 新上传清单(13 个或更少文件)
- 新 GitHub 现读协议(指向新产品主线)

**SK 主仓库(操作系统)只有一个,SKGPT 适配层(项目)可以并行多个**。
