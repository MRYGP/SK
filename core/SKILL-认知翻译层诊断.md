## 一句话

D029 告诉你客户没采用是因为某个因子断裂。本 SKILL 在因子 2(认知映射)和因子 5(系统边界)的具体执行层,拆出"信道哪一环失真",并给四种修复原型。**这是 D029 的"显微镜",不是替代理论**。

---

## 与 D029 的分工

| 用 D029 | 用本 SKILL |
|---------|-----------|
| 判断客户没采用产品卡在哪个因子(1-6) | 已经定位到因子 2 或因子 5,要拆"具体信道哪一环失真" |
| 因子组合矩阵判断产品形态(磁悬浮/工具型/...) | 设计"翻译层"作为具体的修复动作 |
| 跨场景的因子定位 | 单场景的信道分析 |

**判断规则**:先 D029 定位卡点,再用本 SKILL 拆细。**不要跳过 D029 直接用本 SKILL**——那会让你在错的因子上做正确的事。

## 核心模型:发送-编码-信道-解码-接收

```
发送端意图 → [编码] → 信息载体 → [解码] → 接收端认知
```

任何"产品价值没传到客户那边"的场景,可以拆成五个环节。失真发生在两处:

- **编码失真**:发送端以为自己说清楚了,其实没说清楚
- **解码失真**:接收端以为自己听懂了,其实理解错了

**翻译层 = 在编码端和解码端之间补的中间环节**。

---

## 五个失真信号(诊断用)

只要出现以下任一信号,信道里有失真:

1. **高跳出 / 高流失 + 低投诉**——用户不声不响走,不反馈("Spangle 场景:广告点击后 60-90% 跳出,但没人告诉你'页面和广告对不上'")
2. **反复解释但对方仍 get 不到**——你在同维度加量,对方需要换维度翻译("飞书场景:讲完一小时多维表格,客户仍问'这对我有什么用'")
3. **用户行为与口头偏好背离**——嘴上说想要 A,实际选 B
4. **行业必然出现中介/代理/代运营**——原生系统失真严重需要人工翻译层("跨境电商必然出现大量 Shopify 代运营")
5. **懂的人觉得简单,不懂的人觉得复杂**——专业术语 + 隐含假设没翻译

## 断层定位矩阵(因子 2 与因子 5 在这里相遇)

|  | 编码端清晰 | 编码端模糊 |
|---|---|---|
| 解码端有能力 | ✅ 高效传递 | ⚠️ 信息过载 |
| 解码端无能力 | 🔴 **翻译层缺失**(机会区) | ❌ 双向混沌(不可救) |

产品机会只在右下角(🔴):编码端其实清晰,解码端无能力,中间缺一个自动翻译层。

**对应 D029**:右下角同时触发因子 2(解码失败)+ 因子 5(物理摩擦让解码端无法跨越)。这是 D029 的"高因子2 + 低因子5"格的具体形态。

## 四种翻译层原型(设计用)

定位完断层后,从四种原型里选一种或组合:

### 原型 A·参数映射器(Parameter Mapper)

读取输入参数,自动匹配输出。**适用**:输入输出有明确对应关系。**SK 内部样本**:Spangle 读取 UTM 参数 → 匹配商品/文案。**技术形态**:规则引擎 + 轻量 AI。

### 原型 B·场景可视化器(Scenario Visualizer)

把抽象功能翻译成"用了之后会变成什么样"。**适用**:产品功能难被潜在客户想象。**SK 内部样本**:口腔笑容预演 / 植发术前预演 / 家居场景镜 / 医美前后对比照。**技术形态**:模板化内容生成(视频 / 图文)。

### 原型 C·渐进式引导器(Progressive Onboarder)

不一次性塞全部信息,按解码能力逐步释放。**适用**:解码端能力差异大(新手 vs 专家)。**SK 内部样本**:Anthropic onboarding 30 秒诊断问卷。**技术形态**:状态机 + 自适应内容。

### 原型 D·反馈校准器(Feedback Calibrator)

让接收端反应实时回传到发送端,闭环校正。**适用**:发送端愿调整但不知解码效果。**SK 内部样本**:A/B 测试工具 + 客服实时话术提示。**技术形态**:数据闭环 + 实时推荐。

---

## 三问验证(防伪需求)

设计完翻译层,用这三问硬校验:

1. **消失测试**:去掉翻译层,接收端是否立刻回到困惑?如果用户仍无感 → **翻译层是伪需求**
2. **文盲测试**:完全不懂行业的人经过翻译层后能否做出正确决策?如果仍不能 → **翻译层不够底层**
3. **成本测试**:翻译层成本是否低于失真造成的浪费?如果翻译成本 > 浪费成本 → **商业模式不成立**

## 适用边界

### 适用场景

- 已经用 D029 定位卡点在因子 2 或因子 5
- 单次具体的"产品功能 → 客户认知"传递场景诊断
- 产品 onboarding / 落地页 / 销售话术 / 销售物料的设计与诊断
- 任何"发送方说清楚了但接收方没接住"的具体环节拆解

### 不适用场景

- **未做 D029 定位之前**(跳过 D029 直接用本 SKILL = 在错的因子上做正确的事)
- **跨场景理论推演**(那是 D029 的工作,不是本 SKILL)
- **D029 已经能解释清楚的场景**(本 SKILL 是显微镜,不是 D029 没说的额外补充)
- **B 端复杂决策链**(本 SKILL 假设单一接收端,但 B 端有"采购、使用、付款"多个角色,需先用其他工具拆角色)

## 防 AI 高仿陷阱(本 SKILL 自身的元教训)

本 SKILL 是 2026-05-13 由 Kimi 在帮杨推演 Spangle 中国版 + 复用飞书推演时总结出来的"通用框架"。**初稿放在 theory/ 是一个错误**——它把"AI 总结的通用框架"误当成了"经过 SK 流程验证的独立理论"。

**教训**:

- AI 总结的"看上去通用"框架,通常和 SK 既有理论有大量重叠
- 重叠不等于没用——可能是"既有理论的执行视角"
- **判断标准**:这个框架能不能用既有理论 80% 解释清楚?如果能,它就是 SKILL(执行视角)不是 DOC(独立理论)
- 升级 DOC 的红线:必须有 5+ 独立验证样本 + 与所有相关既有理论的边界划清

本 SKILL 从 theory/ 挪到 core/ 是这条红线的执行。
