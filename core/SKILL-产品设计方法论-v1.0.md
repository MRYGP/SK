# SKILL：从"决定做"到可执行产品的设计方法论

> 版本：v1.0（基于老年AI管家产品设计实战提炼）
> 适用场景：推演SOP结论为"做"之后，怎么把想法变成可执行的产品
> 前置条件：已完成推演SOP（v1.2），已确立北极星
> 输出物：产品设计文档（可迭代）+ 技术评估清单 + MVP方案

---

## 与推演SOP的关系

```
推演SOP → "值不值得做？" → 结论："做" → 北极星确立
                                            ↓
                        本方法论 → "怎么做？" → 产品设计文档
```

推演SOP回答"做不做"。本方法论回答"怎么做"。

北极星是桥梁——推演时确立，设计时作为每个决策的过滤器。

---

## 第一步：确认北极星已就位

在做任何产品设计之前，确认北极星文档存在且清晰。

检查：
- 北极星是否一句话？（不超过20字）
- 是否说清了为谁、做什么、结果是什么？
- 是否不包含任何具体功能或形式？

**如果没有北极星，不要开始产品设计。** 先回推演SOP第九步。

本次实战：
> 北极星：让老年人用上AI，从而生活变得更好。

---

## 第二步：综合所有讨论源

一个产品想法可能经历了多轮讨论、跟不同人/AI的对话。在做产品设计之前，先把所有讨论源整理清楚。

**操作：**
1. 列出所有讨论源（跟谁聊了什么、产出了什么文件）
2. 每个讨论源提取"独特贡献"（这次讨论给出了什么其他讨论没有的东西）
3. 做一个对照表，找出共识和分歧
4. 分歧处标记为"创始人判断点"

**本次实战：**

| 讨论源 | 独特贡献 |
|--------|---------|
| Claude推演（29章） | 竞品分析、裂变机制、白标模式、商业模型 |
| KIMI讨论（9章） | "意大利鞋"等式、双轨成就、普朗克陷阱、网络涌现 |
| 第三轮讨论 | 羽毛球原则、亚马逊思维、"生活真的变好" |

分歧示例：Claude侧重"社交货币驱动"，KIMI侧重"存在感驱动"→ 创始人校准后融合为"虚荣是入口，真实价值是留存"。

---

## 第三步：定义"解决什么问题"

这是产品设计最重要的一步。说不清楚解决什么问题，后面全是空中楼阁。

**操作：**
1. 列出所有候选答案（从讨论中提取）
2. 每个答案检验：谁付费？为什么付？天花板多高？
3. 选一个最强的，或组合

**检验标准：**
- 能不能用一句话说清楚？
- 用户听了会不会说"对，这就是我的问题"？
- 这个问题是持续存在的，还是解决一次就没了？

**本次实战的演变：**
```
v0.1："老人需要存在感" → 太抽象，没人为"存在感"付费
v0.2："老人用不上AI，但用上了生活会变好" → 具体、可行动、持续存在
```

**关键纠偏（羽毛球原则）：**
入口动机和产品价值可以不同。老人来的动机可能是"炫耀"（虚荣），但留下的原因是"生活真的变好了"（真实价值）。产品设计要同时满足两者：入口要有吸引力，内核要有真实价值。

---

## 第四步：设计功能层次（递进式价值）

不要平铺所有功能。按"从最容易感知到最深层"排列，形成递进。

**原则：**
- 第一层必须是"30秒就能感受到价值"的东西
- 每一层都独立有价值（不是"必须走到第五层才有用"）
- 用户走到哪一层，取决于他自己，不是产品强制设计的
- 走得越深，生活改变越大

**本次实战：**
```
第一层（AI创意内容）→ "有趣" → 吸引人来
第二层（AI生活助手）→ "有用" → 解决真实问题
第三层（主动关怀）  → "懂我" → 建立情感连接
第四层（社交连接）  → "帮我维护朋友" → 深化真实关系
第五层（反窄化）    → "一直有新东西" → 持续发现可能性
```

**每加一层功能时，问北极星：这一层是让用户的生活更好，还是让产品更复杂？**

---

## 第五步：设计增长引擎

**核心原则：增长不是设计出来的，是价值溢出来的。**

操作：
1. 从推演SOP第六步的增长机制设计中，选出最可能的裂变路径
2. 检验：这个裂变行为是用户"自然想做的事"，还是"被激励才做的事"？
3. 产品只需要降低裂变的摩擦力，而不是创造裂变的动力

**检验标准（亚马逊原则）：**
- 用户留下来的原因是"这里最好"，不是"走不了"
- 免费版本身就好用，付费是"更好"不是"才能用"
- 如果产品不做任何裂变设计，用户自己会不会告诉朋友？

本次实战的纠偏：
```
v0.1：复杂的免费额度+分享解锁+付费解锁机制
v0.2：免费版就很好用，裂变靠"老人忍不住告诉朋友"
```

---

## 第六步：设计商业模型

**操作：**
1. 从推演SOP第七步的商业模型中提取收入来源
2. 算单位经济：每用户成本 vs 每用户收入
3. 确认"用户觉得值"而不只是"我们觉得能赚"

**价值观校准节点：**

在商业模型设计完成后，做一次检查：

| 问题 | 诚实回答 |
|------|---------|
| 这个定价，用户觉得"值"吗？ | |
| 如果成本降低了，会降价吗？ | |
| 有没有"用户不知道但在多付钱"的设计？ | |
| 免费版是真的好用，还是故意阉割逼人付费？ | |

---

## 第七步：处理灵感（灵感纪律）

产品设计过程中会不断冒出新灵感。灵感很兴奋，容易立刻写入设计文档。

**规则：**
1. 记录灵感（不要丢）
2. 标记为"灵感·待验证"
3. 列出验证方式
4. 不写入产品设计文档正文，放在附录或单独文件
5. 验证通过后才合入产品设计

**本次实战：**
"数字人视频通话"的灵感 → 标记为灵感 → 列出技术评估清单 → 设计小步快跑验证路线 → 单独文件存储 → 不改产品设计文档v0.2

---

## 第八步：设计MVP（小步快跑）

**原则：不要一步到位做最终形态。每一步只验证一个假设。**

**操作：**
1. 列出所有X级假设（从推演和产品设计中提取）
2. 按"如果这个假设不成立，后面全白做"的优先级排序
3. 设计Step 0 → Step 1 → Step 2 → ... 的迭代路径
4. 每一步有明确的"继续/停止/回退"判断标准
5. 每一步都能独立运行（即使后面的步骤做不了，当前步骤也是一个有价值的产品）

**检验标准：**
- 每一步是否只验证1-2个假设？
- 每一步之间是否间隔1-2周？（不能太长，失去节奏）
- 每一步是否有明确的"如果失败怎么办"？
- 最终形态的路线图是否清晰但不僵化？

**本次实战：**
```
Step 0（纯文字）→ 老人愿不愿意跟AI聊？
Step 1（语音）  → 语音是否更好？
Step 2（数字人短视频）→ 老人对"有脸的AI"什么反应？
Step 3（实时视频通话）→ 值不值得额外成本？
```

每一步失败就停在上一步。停在语音也是好产品。

---

## 第九步：用北极星过滤全部设计

产品设计文档初稿完成后，逐条过滤：

**对每个功能问：** "这个功能是让用户的生活更好，还是只是让产品更炫？"

**对每个增长设计问：** "这是用户自然想做的事，还是我们设计来逼他做的？"

**对每个商业模型问：** "这是服务用户的钱，还是利用用户的钱？"

**对每个技术选型问：** "这是让用户门槛更低，还是让我们显得更高级？"

**通过过滤的留，不通过的删。**

## 附加原则 · 产品交互的85%原则

**来源**：CEET提炼-三条可用斧头（原始出处：Bjork 1994 意欲困难理论 + CEET H-CET-03）

### 核心原则

产品交互的用户成功率决定了产品的长期命运：

| 用户成功率 | 状态 | 后果 |
|----------|------|------|
| 99%+ | 太顺 | 用户爽但没有成长感，用完即走 |
| 50%-85% | 最佳挑战 | 用户感到"我有成长"，长期黏性 |
| < 50% | 太难 | 用户挫败离开 |

### 应用到AI诊断类产品

**典型错配**：
- 全自动给出诊断结果 → 99%成功率，用户爽但不长记性
- 丢一堆数据让用户自己看 → <50%成功率，用户挫败
- **正确做法**：AI引导用户自己得出结论（AI做80%，用户做20%）

### 检验问题

> **"用户第一次用完我的产品时，是觉得'全自动好爽'，还是觉得'我刚刚学会了什么'？"**

前者 → 要重新设计交互
后者 → 在85%区间，继续迭代

### 和其他原则的关系

- 85%原则 = AVP（C6）的落地实现方式之一
- 99%成功率的产品 = 认知外骨骼 = 用户能力会退化
- 85%成功率的产品 = 有益摩擦 = 用户能力会增长
