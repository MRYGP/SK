## 使用说明

1. **任何项目进入视野时**（拆解别人、评估自己、朋友带想法来聊），都按本清单逐条过
2. **不是读一遍，是逐条回答**——跳过任何一条 = 给自己埋雷
3. **顺序不能颠倒**：先反向（排除失败），再正向（寻找可能），最后身份（照镜子）
4. **任何一条 Part A 触发 → 立即否决，不进入 Part B 之后**

## Part A · 10秒排除线（触碰任何一条 → 直接否决）

## Part C · 正向审问

### C1 · 好奇心 vs 飞轮三问（源自010拆解）

> **用户第一次来之后，还会不会再回来？**

- [ ] **C1-Q1**：好奇用完之后，用户还有没有理由回来？
- [ ] **C1-Q2**：假设每个用户只来一次，这笔生意赚不赚钱？
- [ ] **C1-Q3**：那个会留下来的用户，你见过他本人吗？他叫什么？他上周做了什么？

**检验句**：这三个问题任何一个答不出来或含糊 → 这个项目停在"好奇心层"，没到"飞轮层"。

### C2 · 种树种菜双轨自检（源自D018补丁）

- [ ] **C2-Q1**：这件事本身是菜还是树？
  - 菜的特征：3-6个月能见到第一笔真实现金流，投入产出线性可见
  - 树的特征：需要2-5年积累才能产生复利，护城河随时间增强
- [ ] **C2-Q2**：如果是树，有没有配套的菜地输血？
- [ ] **C2-Q3**：如果是菜，它的产出能为未来某棵树输血吗？
- [ ] **C2-Q4**：有没有陷入两种错配？
  - 错配A：把菜当树浇（一块菜地刚有苗头就all in讲平台故事）
  - 错配B：把树当菜吃（有潜力的树为了短期现金流被砍枝变现）

### C3 · 诊断空白四条件（源自011 MYHAIR AI拆解）

- [ ] **C3-Q1**：高焦虑的人是不是很多？
- [ ] **C3-Q2**：真正在行动的人是不是很少？（否 → 这是红海，不是诊断空白）
- [ ] **C3-Q3**：中间是不是隔着一个"没人回答的问题"？（否 → 已经有人在解这道题）
- [ ] **C3-Q4**：B端有没有人愿意为"已经准备好的客户"付钱？（否 → 没有下游买单方）

**四条件齐备 → 是真正的诊断空白**
**缺一条 → 不是诊断空白，是红海或伪机会**

### C4 · MTP五标准（如果这是一个"想做的"项目）

- [ ] **C4-S1**：能不能让读者5分钟看懂？
- [ ] **C4-S2**：有没有足够信息让读者能判断真假？
- [ ] **C4-S3**：有没有自我攻击点？（允许读者反驳）
- [ ] **C4-S4**：能不能对接具体资源人？
- [ ] **C4-S5**：能不能3个月见到现金流信号？

## Part D · 身份审问（照镜子）

> 这一部分最难。也最重要。

- [ ] **D-Q1**：**"如果这是一个陌生创业者发给我的BP，我会怎么拆它？"**
  - 换个名字（比如叫"某位博士的AI项目"），自己拆一遍
  - 如果换名字之后立刻发现致命问题 → 你在用自我认同回避判断
  
- [ ] **D-Q2**：**"这个项目和那位亏200万的博士，最像的5个地方是什么？"**
  - 必须写出来，写不出5个说明对照不够诚实
  - 相似点 ≥ 3个 → 高风险
  
- [ ] **D-Q3**：**"我在用'聪明'回避哪三个问题？"**
  - 010里提炼：死的创业者都是在用聪明回避关键问题
  - 自己审问：我这次有没有用复杂叠加、漂亮叙事、诱人愿景在**绕过**某个根本问题？
  
- [ ] **D-Q4**：**"我现在的姿势，是'知道自己可能错的人在公开邀请反驳'，还是'一个非常聪明的人在讲一个他相信自己一定能讲成的故事'？"**（源自KAIROS拆解）
  - 后者 = 危险信号

## 六、清单的扩展机制

### 6.1 三层结构

```
草稿区（零摩擦捕获）
  ↓ 出现3次
候选区（结构化整理）
  ↓ 经过实战验证
正式清单（本文档）
```

### 6.2 三种触发器

**触发器1：每次审问完一个项目**
- 问自己："这次用到了清单里没有的判断吗？"
- 如果有，立刻写到草稿区

**触发器2：每次深度对话结束**
- 问自己（或问AI）："这次对话冒出了哪些我以前没想过的角度？"
- 立刻写到草稿区

**触发器3：每月最后一天回看**
- 草稿区 → 哪些出现3次了？升级到候选区
- 候选区 → 哪些被实战验证了？升级到正式清单
- 正式清单 → 哪些3年没触发过？下沉到历史教训库

### 6.3 三次验证规则

一条教训**必须满足以下三次触发**，才能从草稿区升级到候选区：

1. **发现**：第一次在对话/思考中冒出
2. **被引用**：第二次在另一个不同场景被用上（说明不是孤例）
3. **救了一次决策**：第三次在具体项目审问中起了作用（说明有实战价值）

只有满足三次的教训，才值得进入候选区的结构化整理。

**再经过实战验证**（有项目套用它并得到好结果），才能升级到正式清单。

### 6.4 清单条目的标准字段

进入正式清单的每一条，都要有：
- **编号**（A01-A0X / FM0XX / C-Q0X / D-Q0X）
- **一句话表述**
- **检验问题**（能套在任何项目上的那个具体问题）
- **对应FM或理论**
- **来源**（哪次对话/哪个案例提炼的）
- **状态**（v1.0自带 / vX.X新增）

### 6.5 版本变更原则

- 每次升级一条，版本号+0.1（v1.0 → v1.1 → v1.2...）
- 大改结构时，版本号+1.0
- 每次变更必须在Part G留痕
