# SKILL · 下田原则

## 一、核心命题

**在田埂上推演,永远看不见根系真实的样子。**

任何用框架/案例/推演评估"未来能不能长起来",本质上都是
**用已知信息预测未知信息**——而 SK 反复证明:**真正的根系总是
在下田后才涌现**。

## 二、对治什么病

### 病 1·根系幻觉
站在田埂上看,某个方向"根系弱"——但下田后才发现:
- 它有"赋能下游代理商"的延伸根系
- 它有"跨行业模式迁移"的横向根系
- 它有"厂家增值服务"的逆向根系

这些根系在桌面推演阶段**全部看不见**,只有下田后才浮现。

### 病 2·框架的反向滥用
SK 的错位框架/根系评估/双轨验证,**适用于评估外部已发生的案例**,
不适用于评估"SK 自己未发生的方向"——因为外部案例已经下田 N 年,
框架是从他们的实战中抽取的;SK 自己的方向还在田埂上,套用框架 =
**用别人的实战结果,推演自己未来的实战可能**。

这是 D030 标尺错配的元层应用:**已下田的标尺 vs 未下田的标尺**。

## 三、判别"在田埂 vs 在田里"

| 行为 | 田埂 / 田里 |
|---|---|
| 看 100 个案例报告 | 田埂 |
| 用搜索引擎+GPT做竞品研究 | 田埂 |
| 用 SK 框架评估候选 | 田埂(评估完后必须下田) |
| 给 5 个真实卖家打电话/见面 | **田里** |
| 自己付费用候选产品 1 个月 | **田里** |
| 写 1 篇假产品的销售文案投放看转化 | **田里** |
| 找潜在合伙人聊 30 分钟 | **田里** |
| MVP 上线给真实用户试用 | **田里** |

**田埂的标志**:不需要离开屏幕,不会被真人拒绝,不消耗对方时间
**田里的标志**:有真实摩擦,有真实拒绝,有真实数据

## 四、铁律

1. **重大方向决策不能在田埂上完成**——必须至少有一次下田动作的数据反馈
2. **框架是田埂的工具,不是田里的工具**——下田前用框架排除明显的坑,
   下田后框架的优先级低于真实数据
3. **"根系评估"是已下田案例的事后归纳,不能用来评估未下田的方向**
4. **Claude / GPT / 任何 AI 给出的"根系评估""势能评估""可行性评估",
   都是田埂判断**——可以参考,不能定调

## 五、SK 评估引擎的修正

原 SK 评估引擎隐含假设:"框架够好 → 评估够准"
修正后:"框架够好 + 至少一次下田反馈 → 评估够准"

具体操作:任何评估流程,在结论之前必须有"下田动作清单"
和"下田数据反馈记录"两栏。**只有框架推演没有下田数据 = 评估不完整,
不算结论**。

## 六、与归纳演绎双轨验证法的关系

- 归纳演绎双轨 = **田埂上能做的最好的事**
- 下田 = **田埂之外必须做的事**
- 两者不互相替代:双轨能识别哪些假设危险,但只有下田能验证假设是否成立

## 七、一句话

**任何在屏幕前完成的判断,都是田埂判断。SK 路径 B 求生存的核心动作,
是减少田埂时间,增加下田时间。**
