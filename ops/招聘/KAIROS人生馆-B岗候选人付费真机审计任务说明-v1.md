# B岗候选人 · 付费真机审计任务说明 v1

## 一、这是什么

我们有一个已经在运行的手机端 H5 互动作品（单文件为主）。**在正式开发合作之前，我们付费购买一次真机诊断。**

- **只做诊断，不改代码。**
- 报酬：项目费面议，按次结算（诊断完成即付，与是否进入后续合作无关）。
- 我们提供：现有 `index.html` 及相关资产、必要的口径说明（哪些文字锁死、哪些元素可动）。

## 二、交付物（六项，缺一不可）

1. **真机测试矩阵**：iOS 微信 ＋ 至少三台不同档位安卓机（列出机型、系统版本、微信版本）。
2. **全程录屏**：从入口到最后一张卡（card-22）完整走一遍，每台机器一份。
3. **问题清单**，每条必须包含：
   - 现象描述；
   - **复现步骤**；
   - 严重度（阻断／影响体验／瑕疵）；
   - **问题归类（关键）**：这是**代码问题**、**资产缺失**、**内容问题**，还是**设备/平台限制**？
   - 覆盖范围至少包括：音频（尤其 iOS 微信音频解锁）、字幕与真声同步、`fixed` 定位与锁字层错位、缓存与资源版本、页面切后台后的暂停/恢复与音频状态。
4. **四个互动点位的实现建议**（只讲方案，不写代码）：翻页循环、输入后逐字退格、逐条浮现、真实 DOM 打字（**不得用视频模拟文字**）。
5. **HTTPS 部署与版本回退建议**：给小范围试看一个稳定可访问地址的方案。
6. **遗留风险与优先级排序**：如果只有两周，你会先修哪三个？为什么？

## 三、硬性禁止（触犯即终止，费用照付但不进入后续合作）

1. **禁止改动任何代码**（本任务是诊断，不是开发）；
2. **禁止建议重写架构或换框架**（Vue/React 重构不在讨论范围）；
3. **禁止改动任何文案一个字**；
4. **禁止把互动延展模块（stage2）与主片合并处理**——两者是独立部分；
5. **禁止用视频或图片模拟承重文字**（屏内文字必须可被逐字校验）。

## 四、我们如何评估这份报告

好的报告的特征：

- **先问约束再动手**：先确认"哪些字锁死、哪些元素可动、用户用什么设备看、怎么算验收通过"；
- 能把**代码故障、资产缺失、内容问题、设备限制**分得清清楚楚，不混为一谈；
- 对错字、字幕错位、音频提前结束**有本能敏感**；
- 交付习惯是"我在这台手机上给你演示"，不是"我本地是好的"。

不合适的信号：

- 开口就建议重构；
- 只批评现有代码，不给兼容性方案；
- 说"微信音频限制没办法"；
- 没有 iOS 和安卓真机验证条件；
- 交一份泛泛而谈的架构报告，没有可复现的问题条目。
