# 实习生A · 王 · 工作手册

> 直属：杨
> 搭档：周（实习生B，技术实现）
> 核心定位：**双角色**——文章流水线执行者 + 产品需求链接者与全品评审者
> 版本：v3.0 · 2026-04-18（新增产品评审角色）

## 零、你是两件事的人（新版核心）

### 角色1：文章流水线执行者（不变）

把杨的底稿变成公众号文章发出去。从改写、检测、排版、发布、冷启动、分发到数据追踪，整条流水线你负责跑。

这部分**和原来完全一样**，没有变化。参考本手册第二节。

### 角色2：产品需求链接者 + 全品评审者（新增）

我们决定开始做最简MVP。方向由杨推演，周动手实现。你在中间：

- **链接者**：作为杨和周之间的辅助沟通者，不是说杨和周不直接对接，而是你作为第三方在场，确保需求不失真
- **全品评审者**：你要听懂杨的产品推理、周的技术方案，**可以从文章读者的视角质疑需求本身**

第二个角色比第一个重要。你不是被动的信息中转员——你是"从读者视角"的产品经理半角色。你见过所有读者的反馈、评论、私信——你比杨和周更懂"真实读者会怎么看这个产品"。

## 一、为什么你能做评审者

你的信息结构特殊：

| 信息来源 | 谁接触最多 |
|---------|---------|
| 读者原话（评论+私信+朋友圈评价）| **王** |
| 跨平台反馈（即刻/知乎/小红书）| **王** |
| 群里的即时反应 | **王** |
| 产品推理和理论 | 杨 |
| 技术实现细节 | 周 |

杨会陷在推理里，周会陷在技术里。**读者视角的校正，只有你能做**。

你的评审标准不是"技术对不对"、也不是"推理严不严谨"，而是：

- **这个产品如果要做一篇文章介绍，读者能不能一句话复述它解决什么？**
- **目标用户（那个卡住的人）真的存在吗？还是只是推理出来的画像？**
- **文章能不能给这个产品钓到用户？如果不能，这个产品方向可能有问题**

## 二、文章流水线（不变）

完整流程保持原样：

```
杨写底稿 → 你标注 → 写作GPTs改写 → 你检查 → 读者GPTs检测
→ 你提交杨审核 → 杨确认 → 你排版发布 → 冷启动 → 分发 → 数据
```

具体操作细节（标注圣物、GPTs话术、排版、冷启动、数据等）**全部延续原手册 v2.0**——不要重写，按老办法做。唯一的变化是以下两点：

**变化1：发法C已确立**

所有文章都要走发法C——结尾要有"如果你是那个卡住的人，加我微信聊"的直接召唤。不是"你怎么看"这种评价式提问。

你检查改写稿时，把"发法C结尾"作为必查项。不合规的打回，让写作GPTs重写。

**变化2：捕捞和筛选的动作延续不变**

发布后 72 小时盯 4 个入口（评论/私信/朋友圈/跨平台），筛出 A/B/C 三档。A档（真实卡住的人）交给杨——具体操作照原流程。

但注意——**捕捞不是产品决策的前提**。即使 A 档一个人都没捞到，产品推进照样不停。捞到就是加速，没捞到我们自己跑。

## 三、产品评审者角色：每周两个动作

### 动作1：产品讨论参会+事后复盘

杨每次做完产品推演（或者要讨论下一步产品方向），**会在群里召集你和周一起讨论**。讨论可能是微信语音、也可能是线下。

你的任务：

**会前**：
- 读杨的推演流水账（至少通读一遍）
- 读对标产品的案例卡（比如 MYHAIR AI 就读 CASE-011）
- 带着三个问题到会场：
  1. "那个卡住的人"的画像是什么？叫什么？哪里的？为什么卡住？
  2. 如果我要写一篇文章介绍这个产品，我的那一刀是什么？
  3. 这个产品跟对标产品，最核心的差异在哪里？（不是表面功能，是逻辑）

**会中**：
- 听懂为主，不打断
- 周讲技术方案时，你不用评价技术本身（那不是你的长处）——但你可以问"这个技术方案，我怎么跟读者解释"
- 如果杨的需求描述太抽象（比如"帮用户快速判断"），你要追问："具体到哪一步？读者看到什么？"
- 如果周的方案听起来脱离了用户场景，你要追问："这个用户拿到手会先做什么？"

**会后24小时内**（这是你的核心交付物）：

写一份**产品需求评审备忘**，交给杨和周。模板：

```markdown
# 产品需求评审备忘·[产品名]·YYYY-MM-DD

## 本次讨论共识
- 选的产品方向：
- 对标产品逻辑：
- 那个卡住的人：[具体到画像]
- 最简MVP是什么：[一句话]

## 我听懂的部分
（复述一遍你听懂的需求，给杨和周校对）

## 我没听懂或感到模糊的部分
（这是最有价值的——你听不懂，读者也听不懂）

## 我作为评审者的3个质疑
1. 关于用户：
2. 关于价值：
3. 关于文章怎么写：

## 下一步动作
- 杨：
- 周：
- 我：
```

这份备忘不是走流程——是你把会议的"共识"显性化，防止杨和周事后理解不一致。

### 动作2：周做MVP过程中的持续沟通桥

周开始动手后，你要保持对进度的关注：

- 每周一次小同步（15分钟），问周"这周做了什么、卡在哪里、需要杨看什么"
- 如果周卡在"这个功能到底要不要有"——把问题整理清楚，带给杨
- 如果周自己往一个方向走，但你感觉"偏了"——先问周"为什么这么做"，再决定要不要让杨介入

**关键原则**：你不是周的审查者，是周的减压阀。周做得顺就不打扰；周迷路了你先帮他找回来，找不回来再找杨。

## 七、核心原则

你现在有两件事：
1. **文章线不断**——每周稳定出文章，保持公众号节奏
2. **做周和杨之间那个问对问题的人**——你没听懂的地方，读者也不会懂；你能质疑的角度，杨和周都看不到

**第二件事比第一件重要**。文章流水线是你现在就熟的，产品评审是你新长出来的能力。前者保底，后者决定你的天花板。
