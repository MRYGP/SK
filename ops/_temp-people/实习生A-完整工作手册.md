## 零、你现在是三件事的人

### 角色1：文章流水线执行者（基础保底）

把杨的底稿变成公众号文章发出去。从改写、检测、排版、发布、冷启动、分发到数据追踪，整条流水线你负责跑。

具体操作细节（GPTs话术、排版、冷启动、数据等）按 `content/文章发布SOP.md` 执行，不重复写在本手册。

**变化：发法C已确立**

所有文章结尾必须有"如果你是那个卡住的人，加我微信聊"的直接召唤。你检查改写稿时，把"发法C结尾"作为必查项，不合规的打回。

### 角色2：产品需求链接者 + 评审者

我们在做 SK Workbench 工具链。方向由杨推演，实习生B动手实现。你在中间：

- **链接者**：确保杨的产品需求到B的技术实现不失真
- **评审者**：从文章读者视角质疑需求本身

你是三人里最接近真实读者的人。你见过所有读者的评论、私信、朋友圈反馈——这是杨和B都没有的信息。

你的评审标准不是"技术对不对"，而是：
- 这个产品/工具如果要写成文章，读者能不能一句话复述它解决什么？
- 目标用户（那个卡住的人）真的存在吗？还是推理出来的画像？
- 这个工具能不能帮到我自己的真实工作流？

### 角色3：SKILL 工具验收者（新增）

B 做完工具后，你是第一个真实用户。你不看代码，只看：
- 用户第一步知道怎么用吗？
- 输出能不能看懂？
- 能不能帮助真实工作流（文章发布 / 招募私信 / 仓库审计）？
- 这个工具我愿意每次都用吗？如果不愿意，为什么？

**第三个角色决定 B 的工具有没有用**。工具做出来没人用等于没做。

## 二、文章流水线

完整流程：

```
杨写底稿 → 你标注 → 写作GPTs改写 → 你检查 → 读者GPTs检测
→ 你提交杨审核 → 杨确认 → 你排版发布 → 冷启动 → 分发 → 数据
```

发布后 72 小时盯 4 个入口（评论/私信/朋友圈/跨平台），筛出 A/B/C 三档：
- A 档（真实卡住的人）→ 截图原话发给杨，附一句话说明"这个人可能是[什么身份]，[什么意图]"
- B 档（有兴趣但没出手）→ 记入 `ops/读者反馈汇总.md`
- C 档（路人赞赏）→ 记入即可，不追踪

注意：**捕捞不是产品决策的前提**。A 档没有也不停产品推进，捞到就是加速。

## 三、产品评审者角色：每次产品讨论后的两个动作

### 动作1：产品讨论参会 + 24小时内产出评审备忘

杨每次做完产品推演，**会召集你和实习生B一起讨论**（微信语音或线下）。

**会前**：
- 读杨的推演流水账（至少通读一遍）
- 读对标产品的案例卡
- 带三个问题到会场：
  1. "那个卡住的人"是谁？哪里的？为什么卡住？
  2. 如果要写一篇文章介绍这个方向，我的那一刀是什么？
  3. 这个方向跟对标案例，最核心的差异在哪里？

**会中**：
- 听懂为主，不打断
- B 讲技术方案时，不评价技术本身，可以问"这个方案我怎么跟读者解释？"
- 如果杨的需求太抽象（比如"帮用户快速判断"），追问："具体到哪一步？用户看到什么？"
- 如果 B 的方案脱离用户场景，追问："这个用户拿到手会先做什么？"

**会后24小时内**（核心交付物，主动产出，不等杨提醒）：

```markdown
# 产品需求评审备忘·[产品/方向名]·YYYY-MM-DD

## 本次讨论共识
- 方向：
- 那个卡住的人：[具体到画像]
- 最简 MVP 或第一个 SKILL Runner 是什么：[一句话]

## 我听懂的部分
（复述一遍你听懂的需求，给杨和B校对）

## 我没听懂或感到模糊的部分
（这是最有价值的——你听不懂，读者也听不懂）

## 我的3个质疑
1. 关于用户：
2. 关于价值：
3. 关于文章怎么写 / 工具怎么用：

## 下一步动作
- 杨：
- B：
- 我：
```

**这份备忘的目的**：把会议共识显性化，防止杨和B事后理解不一致。不是走流程，是真正把"我们说好的是什么"写下来。

### 动作2：B 开发过程中每周 15 分钟同步

B 开始做 SKILL Runner 后，你要保持对进度的关注：

- 每周一次 15 分钟小同步（微信或线下），问 B："这周做了什么、卡在哪里"
- 如果 B 卡在"这个功能到底要不要有"——整理清楚，带给杨
- 如果 B 自己往一个方向走但感觉"偏了"——先问 B "为什么这么做"，再决定要不要让杨介入
- 你不是 B 的审查者，是 B 的减压阀。B 做得顺就不打扰；B 迷路了先帮找回来，找不回来再找杨

## 四、SKILL 工具验收：每次 B 交付后

B 每完成一个 SKILL Runner 版本，你要做验收：

**不看代码，只用工具。** 按以下标准逐条判断：

| 验收项 | 通过标准 | 失败时怎么做 |
|--------|---------|------------|
| 第一步能不能开始 | 我不看 README 也知道先做什么 | 告诉 B "我第一步卡在哪里了" |
| 输出能不能看懂 | 我不解释也能判断这条是问题还是正常 | 截图给 B，指出哪里看不懂 |
| 这条规则值不值得自动化 | 这条节省了我超过3分钟/篇的时间 | 告诉 B 建议删掉这条规则 |
| 有没有漏报 | 我手工能抓到的问题工具也抓到了 | 告诉 B 漏掉了什么，为什么漏 |
| 有没有误报 | 没问题的文章工具没有乱报 | 告诉 B 误报了什么，可能原因 |

验收完产出一份报告（格式见 `ops/QA工具验收报告-v0.1.md`）。**每次验收报告必须有"下一版最需要改的是哪一条"**。

## 九、核心原则

你现在有三件事：
1. **文章线不断**——每周稳定出文章，保持公众号节奏
2. **做杨和B之间问对问题的人**——你没听懂的地方，读者也不会懂；你能质疑的角度，杨和B都看不到
3. **做 SKILL 工具的第一个真实用户**——你愿不愿意用，决定工具有没有做对

**第二件和第三件比第一件重要**。文章流水线是你熟的，评审和验收是你新长出来的能力。前者保底，后者决定你的天花板。
