## 一、为什么需要这个候选方法

SK 现有清单擅长排雷、识别伪需求、阶段错配和失败模式。

但在战略级节点上，还缺一类问题：

> 当行业惯例本身可能是错的，如何把问题拆回物理 / 经济 / 组织约束，并设计可控失败来验证新组合？

本方法只用于战略级节点，不用于日常运营决策。

适用：

- 技术架构选型
- 定价模型设计
- 冷启动路径规划
- MVP 验证设计
- 资源分配与并行推进
- 判断某个方向是否值得“炸一次”

不适用：

- 客服话术
- 当日投放素材
- 普通选题润色
- 已经明确进入执行期的小优化
- 没有失败预算的小团队盲目多线作战

---

## 二、核心定义

极限约束推演法不是“创新”。

它是：

> 把行业惯例拆到不可再分的物理、经济、监管、组织约束，再重新组装出一个可以被小规模验证的方案。

它包含四个动作：

1. 第一性拆解：拆到底层约束；
2. 并行试错：把可并行的验证线程拆开；
3. 失败预算：预先定义允许炸掉什么；
4. 叙事折现：判断叙事能否换来当下资源。

## 三、七个审问问题

### Q1 · 第一性约束

如果完全不参考行业惯例，这件事的物理 / 经济 / 监管 / 组织约束分别是什么？

检查：

- 哪些成本是真成本？
- 哪些成本是惯例、渠道、组织、信任成本？
- 哪些约束不可绕？
- 哪些约束只是旧玩家不愿改？

---

### Q2 · 理论成本下限

当前行业价格里，哪些是不可避免成本，哪些是惯性成本？

检查：

- 原材料 / 推理 / 人工 / 履约成本是多少？
- 渠道加价是多少？
- 信任成本是多少？
- 用户真正为哪一部分付费？

输出：

> 理论下限成本：
> 当前市场价格：
> 中间差额来自：

---

### Q3 · 并行试错

这个项目哪些环节可以并行推进？哪一条线失败会拖死全局？

可并行：

- 产品原型
- 种子用户访谈
- 合规检查
- 内容招募
- 渠道测试
- 成本测算

必须串行：

- 依赖同一关键假设的验证
- 同一资金池无法同时承受的试错
- 一个失败会让全局无法继续的动作

硬规则：

> 没有 48 小时熔断机制的并行，不叫时间杠杆，叫失控。

---

### Q4 · 失败预算

这次试错允许炸掉多少钱、多少时间、多少人力？

必须写清：

| 项 | 预算 |
|---|---|
| 时间 |  |
| 现金 |  |
| 人力 |  |
| 可接受归零方向 |  |
| 不可接受损失 |  |

关键判断：

> 单次失败必须换回边际信息增益。
> 如果第二次失败和第一次失败原因相同，就是浪费。

---

### Q5 · 叙事折现

这个叙事能换来什么当下资源？

可折现资源：

- 用户预付款
- 合伙人信任
- 渠道合作
- 供应商让利
- 试点机会
- 数据权限
- 时间窗口
- 内容关注

危险信号：

- 只能换来自我兴奋；
- 只能换来点赞；
- 不能换来任何真实资源；
- 用未来愿景掩盖当前没有交付；
- 没有里程碑却提前讲终局。

判断句：

> 没有交付记录的宏大叙事，不是资本化，是透支信用。

---

### Q6 · 组织硬度

这个“不可能目标”需要什么人才密度？

检查：

- 2 个顶级人能不能胜过 5 个普通人？
- 创始人是否具备技术 / 行业可信度？
- 团队有没有足够自主权和试错预算？
- 目标极端，资源是否也极端支持？
- 如果只是高压，没有资源支持，就是官僚式压榨。

---

### Q7 · 不能学清单

以下不许普通小团队照抄：

| 模式 | 为什么危险 |
|---|---|
| 个人品牌即公司估值 | 早期创始人没有长期交付信用，容易先透支信任 |
| 多线作战 | 资源不足时多线 = 全线失血 |
| 极限工作伦理 | 创始人内驱不能直接转嫁给团队 |
| 监管套利 | 小团队没有法务、资本和公共关系能力消化风险 |
| 未来护城河 | 把终局当起点，容易触发阶段错配 |
