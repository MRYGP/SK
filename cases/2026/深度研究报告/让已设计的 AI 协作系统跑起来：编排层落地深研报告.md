## 连续发布的内容生产流水线

你当前的瓶颈不是「写得出来」，而是「在多工具、多角色之间稳定流转」。要把 001-003 连发跑起来，关键不是再叠工具，而是明确 **单一事实源（SSOT, single source of truth）** 与 **最短可用的交接协议（handoff protocol）**，把“复制粘贴”从主流程里剥离出去，变成可替换的操作模块。

### 1.1a 哪些环节可以替代手动复制

#### Claude Project 之间能不能自动传递内容？

（已验证事实）Claude 的 Projects 机制是“自包含工作区”：每个 Project 有独立的聊天历史与知识库。也就是说，**Project 与 Project 之间没有内建的“自动共享上下文/文件”机制**，默认就是隔离的。citeturn15search1

（已验证事实）Claude Projects 的知识使用方式是 RAG（检索增强生成）：Claude 会从 Project 知识库中检索相关片段带入当前对话，而不是把所有文件完整塞进上下文；官方也强调这能让你“探索比上下文窗口更大的内容”，并提到可处理远超上下文的材料规模（例如“10 倍以上”这一类表述）。citeturn1search3

**结论**：  
- **“Claude Project → Claude Project 自动传递”**：在 Claude 网页产品层面，缺少你要的“自动联通”；最现实的是把交接放到外部中间层（文件库/表格/仓库），或转 API/Agent 编排实现。citeturn15search1turn1search3  
- **你要的“编排层”更像是：把 Project 当执行节点，把外部中间层当“共享大脑/总线”。**（[推测]）

#### ChatGPT GPTs 与 Claude Projects 有没有自动联通方案？

（已验证事实）OpenAI 已将 **Model Context Protocol（MCP）** 作为 ChatGPT Apps / deep research / API integrations 的扩展方式：你可以搭建 MCP server，把工具与数据源接入 ChatGPT。citeturn11search0  
（已验证事实）ChatGPT 侧已出现“自定义连接器（MCP）”的产品化路径（在发布说明中可见“Custom connectors via MCP”等表述）。citeturn11search2  
（已验证事实）Anthropic 侧也提供 Claude Desktop 的 MCP 扩展/本地 MCP server 接入指南，并解释了安装自定义扩展的入口与组织级扩展机制。citeturn11search8turn11search27  
（已验证事实）Claude Code 也明确支持通过 MCP 连接外部工具与数据源。citeturn11search31  

**结论**：  
- **“ChatGPT GPTs ↔ Claude Projects”** 没有“产品层一键互联”，但 **“ChatGPT（通过 MCP）↔ 共享数据源 ↔ Claude（通过 MCP/Claude Code）”** 在工程上是可行路线。citeturn11search0turn11search8turn11search31  
- 对你来说，最可控的共享数据源不是再买一个 SaaS，而是你已经拥有的 **E:\sk + GitHub 仓库**（你可控、可审计、可回滚）。（[推测]）

#### 有没有人搭过类似“多 AI 工具串联”工作流？用什么编排工具？

你要的“多节点串联”在 2025-2026 的主流路径大致分三类（按落地难度从低到高）：

**A. 浏览器插件/本地工具把“发布分发”这个最痛的复制粘贴点直接干掉**  
（已验证事实）开源 Chrome 扩展 **Wechatsync（文章同步助手）** 支持把微信公众号文章“一键同步到 29+ 平台”，明确覆盖知乎、小红书等；并强调所有操作在本地浏览器完成、Cookie/账号信息不经第三方服务器，且支持 MCP 与 Claude Desktop/Claude Code 集成。citeturn20view1  
这类工具的价值是：你不需要先解决“跨 AI 工具传递”，就能立刻把“发布分发”段的复制粘贴从主流程中移除。

**B. 低代码/开源自动化编排（n8n / Make / Zapier / 飞书工作流）**  
（已验证事实）n8n 提供 Anthropic Chat Model 集成，并强调可对接大量应用做自动化。citeturn19search19  
（已验证事实）n8n 官方也在 2026 讨论“AI Agents 上生产”的部署最佳实践。citeturn19search12

**C. Agent 驱动编排（OpenClaw / Claude Code + MCP / ChatGPT MCP 连接器）**  
（已验证事实）OpenClaw 是可本地运行的个人 AI assistant，支持大量通讯/协作渠道（含飞书等），并可通过 Skills 扩展能力。citeturn12search4

这类方案可以把“复制粘贴”进一步变成“自动写入共享文件/表格/任务系统”。

#### 中国网络环境下哪些编排工具可用且稳定？

这里的“稳定”要拆成两件事：  
1）访问稳定（不依赖经常波动的跨境网络）；2）执行稳定（错误可追踪、可重放、可回滚）。

（已验证事实）飞书多维表格已经把“AI 生成文本”“发送 HTTP 请求”“发送消息”“条件判断/多分支/循环”等自动化/工作流能力产品化，且主打“无需懂代码”。citeturn21view3turn23search12

相对地，Zapier/Make 这类海外自动化平台在中国的“访问稳定性”更不确定（数据不足；未找到官方对中国大陆可用性的承诺来源）。

### 1.1b 冷启动执行中，AI 能接管哪些重复环节？

你列的 5 项（知乎改编、即刻、小红书 6 张图文卡、朋友圈、群发引导语）可以按“对平台语境依赖程度”分成两层：

#### 可以由 AI 直接出终稿（实习生复制粘贴即可）

**朋友圈文案**：高度模板化、短文本、容错高。只要你把“行动号召/不含招募段落/语气克制”写进模板，AI 可直接给终稿。（[推测]）

**即刻发帖**：也偏短文本，适合“观点 + 反常识 + 一句话结尾”结构；AI 给到终稿，实习生主要做 1 次语气微调（更像真人）。  
数据不足：未找到“即刻官方/权威”对 AI 批量发布的最佳实践说明。但从内容形态推断可行。（[推测]）

#### AI 出初稿 + 人工微调更稳

**知乎改编**：知乎更吃“论证链 + 反驳预期质疑 + 结构化小标题”。AI 可以把公众号文改成问答体并补论证，但你最好让实习生做两件事：  
1）把“仿真案例/数据”标注来源，避免虚构；2）把“结论口吻”调成更克制。（[推测]）

**群发引导语（不同群不同话术）**：群差异太大，“错一句”成本高。AI 适合生成 3 档话术（冷/中/热群），实习生根据群气质与禁忌词做最后改写。（[推测]）

**小红书 6 张图文卡片**：AI 可以产出“6 页分镜式文案”（每页 10-30 字 + 页标题 + 视觉提示），但真正决定传播的是版式与封面图。这里建议：  
- 文案：AI 主导；  
- 设计：用固定模板半自动；  
- 封面：人工把关（决定点击）。  
（已验证事实）Canva（可画）提供小红书图文模板与在线制作入口，适合“模板化生产”。citeturn10search3  
（已验证事实）飞书多维表格案例中展示了“AI 字段捷径”可用于生成分镜图、并通过自动化做催办与任务提醒，说明“内容生产 + 项目管理”可以在一个表格里串起来。citeturn21view3  

#### 有没有现成工具能一键生成“一篇文章→多平台适配版”？

对你这种“公众号为母体”的矩阵分发，最接近“一键”的现实答案目前是两段式：

- **段 1：从公众号编辑器/网页把内容抽取成统一稿**  
- **段 2：把统一稿同步到多个平台**

（已验证事实）Wechatsync 的功能描述明确覆盖：从公众号编辑器提取文章，一键同步到多平台，并列出支持平台清单；同时提供 CLI 与 MCP server，可被 Claude Code/Claude Desktop 以自然语言调用。citeturn20view1  

这基本能把你流程里“发布后多平台复制粘贴”这一坨最贵的动作直接消掉（尤其是知乎、小红书等）。citeturn20view1  

局限：它是否覆盖你要的“即刻”等平台取决于其适配器列表；在你给的 Wechatsync 支持清单里未看到“即刻”（数据不足）。citeturn20view1

### 1.1c 从 001 发布到持续周更：1 天游得起来的最小可行方案

你的约束是：不额外大投入、当天搭好、实习生可执行。最小可行方案不追求“全自动”，追求 **“一次输入，多处输出；交接可追踪；失败可回退”**。

下面这套我按“今天就能跑”来设计（核心是：**SSOT 文件 + 表格工单 + 一键分发**）。

#### 最小方案架构（不需要写代码）

**SSOT：每篇文章一个文件夹（Git 仓库里）**（[推测]）
```
E:\sk\cases\2026\publish\
  001\
    001-底稿.md
    001-公众号成稿.md
    001-读者压力测试.md
    001-多平台素材包.md
    assets\
  002\...
  003\...
```

**工单：一张飞书多维表格作为“执行台”**（[推测]）  
表里每行对应一篇文章，字段建议：状态、发布日期、公众号链接、知乎稿、小红书 6 卡文案、朋友圈文案、群发话术（冷/中/热）、待办人、验收人、复盘要点。

为什么是飞书多维表格：  
（已验证事实）飞书多维表格/工作流能力包含条件判断、分支、循环、发送消息、发送 HTTP 请求、以及“AI 生成文本”等节点，适合把“内容 → 任务 → 催办 → 回收”做成轻量闭环。（中国相关信息可靠度：高）citeturn21view3turn23search12  

**分发：Wechatsync 作为“多平台同步发动机”**  
（已验证事实）Wechatsync 支持从公众号编辑器提取文章并同步到 29+ 平台，且强调本地执行与开源可审计；并提供 CLI 与 MCP 集成路径。citeturn20view1  

#### 具体步骤（按你 001-003 连发节奏）

**步骤 A：把“写作链”压缩成 2 次交接**（[推测]）  
1）你在主 Project 完成拆解/推演后，输出到 `001-底稿.md`（只做一次复制）。  
2）写作工坊润色后，输出到 `001-公众号成稿.md`（只做一次复制）。

> 这两次复制短期很难彻底消失（除非上 API/Agent），所以最小方案接受它，但把后面所有复制都干掉。

**步骤 B：把“读者链”合并到 ChatGPT 一个项目对话里**  
（已验证事实）ChatGPT 的 GPTs FAQ 明确：你可以在同一对话中用 “@” 调用不同 GPT，而不需要开新对话，从而共享同一上下文。citeturn2search1  
这对你意味着：  
- 第一读者、第二读者、深度研究员（如果都是 GPTs）可以在 **同一个 ChatGPT 项目对话**里轮流 @ 调起；  
- 你只需要粘贴一次文章正文，然后连续拿到三种反馈。  
（已验证事实）同时，该 FAQ 也明确 GPTs 不会使用“保存的记忆 / 自定义指令 / 过去对话”（即 GPTs 自身不具备你想要的长期记忆共享）。citeturn2search1  

**步骤 C：把“冷启动多平台分发”从人工复制变成“一键同步 + 少量手工”**  
1）实习生 A 在公众号后台排版发布（这是不可替代的强人工环节，尤其要人工审美）。  
2）发布后，用 Wechatsync：从公众号文章页/编辑器直接抽取 → 同步到知乎/小红书等支持平台。（已验证事实）citeturn20view1  
3）飞书多维表格里自动生成：朋友圈文案、群发引导语三档、知乎“补论证段落”（AI 字段生成文本）。citeturn21view3turn23search12  
4）即刻若不在 Wechatsync 支持范围内：保留复制粘贴，但内容由表格 AI 字段直接给终稿，人工只做语气微调。（[推测]）

#### 风险与兜底

- 若 Wechatsync 某平台适配失效：你依然可以把 `001-公众号成稿.md` 作为统一稿手动发布，不影响主流程。（已验证事实：Wechatsync 明确“网页转 Markdown、打包 ZIP”等导出能力，提供替代路径）citeturn20view1

#### 本部分「如果只做一件事」

## 六节点共享记忆

你要的不是聊天软件的“记忆按钮”，而是一个能沉淀为资产的 **组织记忆系统**：北极星、判断框架、失败模式、可复用模板、已验证情报、以及每次迭代后的“经验差分”。

### 2.1 各工具当前的记忆能力

#### Claude Projects 的 Knowledge Base（RAG）机制

（已验证事实）Claude Projects 允许你在一个 Project 内上传文件并作为知识库使用；每个 Project 是自包含工作区，有自己的知识库。citeturn15search1turn1search3  
（已验证事实）其知识使用属于检索式：官方强调“contextual retrieval”可以让你访问比上下文窗口更大的内容。citeturn1search3  

**能做到的“记忆”**：  
- 在同一个 Project 内，长期保留一组“参考资料 + 指令”，让 Claude 在对话时检索引用。（已验证事实）citeturn15search1turn1search3  

**局限**：  
- 无法天然跨 Project 共享；你现在“主 Project → 写作工坊”的经验断层，本质就是这个隔离带来的。citeturn15search1  
- 知识库的硬上限（文件数量/总容量/单文件大小）官方公开信息不完整：未找到一份权威页面把“Project 知识库上限”用可核验数字写清楚（数据不足）。

#### Claude 的 Memory 功能（2026-03 最新状态）

（已验证事实）Claude 的“Chat and memory”说明提到：当 Memory 打开时，Claude 可能从你的聊天历史中综合形成记忆；同时明确 **Projects 中的消息不会被用于记忆**（也就是 Project 内对话不会进入全局记忆）。citeturn1search2  
（已验证事实）Claude 也提供“导入/导出记忆”的能力入口，说明记忆具备一定可迁移/可管理性，但这仍是“产品级记忆”，不是你要的“组织知识资产管道”。citeturn1search7  

**结论**：Claude 的 Memory 更像“个人偏好与背景”的轻量层，而你要的“系统记忆”仍需外部中间层。（[推测]）citeturn1search2turn1search7  

#### ChatGPT 的 Memory 功能与 GPTs 共享性

（已验证事实）OpenAI 在 Memory 相关说明中强调：ChatGPT 的 Memory 会让它记住你请求其记住的内容、并在后续对话中参考；并提供控制与开关。citeturn0search7  
（已验证事实）但 GPTs FAQ 明确：自定义 GPT **不会**使用 saved memory / 自定义指令 / 过去对话。也就是说，你做的“第一读者/第二读者/深度研究员”这些 GPTs，默认不共享你在 ChatGPT 主体的记忆层。citeturn2search1  

（已验证事实）OpenAI 的 Projects 提供“project-only memory”这类隔离/聚焦机制，用于把项目上下文集中管理。citeturn1search16  

**结论**：  
- ChatGPT 的 Memory/Projects 能增强“同一工具内的连续性”，但 **无法自动覆盖你跨工具/跨节点的共享记忆目标**。citeturn2search1turn1search16  

#### “主 Project 新洞察”能否自动同步到写作工坊与 GPTs 研究员？

当前“产品内置自动同步”路径：数据不足（未找到 Claude Projects 与 ChatGPT GPTs 之间的官方同步机制说明）。

### 2.2 多工具间共享记忆的可行路径与已有实现

你描述的“E:\sk 作为共享大脑”非常接近 2025-2026 agent 圈的主流共识：**把长期状态外置成文件/数据库，让模型每次从外部重建上下文**。这也与一些研究方向一致（例如强调长期任务中“外置状态”的框架思想）。citeturn24academia17  

**路径一：用本地/仓库文件作为共享大脑（最符合你现状）**  
- 你的 E:\sk 已经是结构化知识库；差的是“把对话产出自动沉淀进去”。（[推测]）

**路径二：用 MCP 把“共享大脑”接到各节点**  
（已验证事实）OpenAI 提供 MCP server 开发/接入文档，目标就是让 ChatGPT/深度研究等能连接外部数据源与工具。citeturn11search0  
（已验证事实）Claude Desktop/Claude Code 同样可通过 MCP 接入本地/外部工具。citeturn11search8turn11search31  

**路径三：OpenClaw 的 Persistent Memory 机制是否能做这件事？**
